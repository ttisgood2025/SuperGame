# 萌宠三叠消消乐（Cocos Creator）标准化开发操作指南（从 0 到当前可运行版）

> 目标：**最少步骤、少踩坑、快速得到可运行版本**，并能稳定迭代到 60 关正式版本。

---

## 1. 项目目标与当前状态

### 1.1 当前已达成状态
- 已有可运行最简玩法：点击牌面 → 进槽 → 三消 → 胜负判定 → 重开/下一关。
- 已有 `QuickStartLauncher` 一键场景初始化，减少 Cocos 端手动拖拽。
- 已有 60 关配置 JSON，可由 `resources` 动态加载。
- 已有失败兜底机制：`resources` 加载失败时使用内置关卡，避免卡死在“初始化中”。

### 1.2 迭代目标（后续）
- 把临时可玩版升级为正式玩法版：障碍机制、引导、特效、美术资源、音频、数据埋点、商业化。

## 2. 从 0 到可运行：最简标准流程（推荐）

### Step A：新建工程
1. 打开 Cocos Creator（建议与开发版本一致）。
2. 新建 2D 项目。

### Step B：导入资源
1. 将仓库中的 `assets/` 整体复制到 Cocos 项目根目录（覆盖同名目录）。
2. 等待资源导入完成。

### Step C：挂载启动器
1. 打开主场景。
2. 选中 `Canvas`。
3. 挂载 `QuickStartLauncher`。

### Step D：运行验证
1. 点击运行（Preview）。
2. 预期结果：
   - 正常看到可点击牌面（不是静态空界面）。
   - HUD 有状态、关卡、槽位信息。
   - 点击牌面后状态变化。

## 3. 架构分层（最小可维护结构）

- `assets/scripts/QuickStartLauncher.ts`
  - 负责“启动期”自动组装：创建控制器、创建基础 UI、绑定 UI 脚本、加载关卡。
- `assets/scripts/core/GameController.ts`
  - 玩法主控制器（状态机、关卡开始、点牌、胜负、事件通知）。
- `assets/scripts/core/*Manager.ts`
  - `BoardManager`：牌堆逻辑。
  - `SlotManager`：槽位与三消。
  - `LevelManager`：关卡数据读取。
  - `EconomyManager`：结算货币。
- `assets/scripts/ui/SimpleBoardUI.ts`
  - 将 `GameController` 数据渲染成可见 UI。
- `assets/resources/config/levels.json`
  - 关卡数据源。

## 4. 历次关键问题与经验总结

## 4.1 好经验（保留）

1. **先跑通最小闭环，再做美术和复杂规则**
   - 先保证核心玩法 loop 是活的，再叠加资源，迭代成本最低。

2. **把“手工绑定”变“程序装配”**
   - `QuickStartLauncher` 自动创建节点和绑定，显著减少 Cocos 编辑器人工出错。

3. **UI 必须监听状态变更事件**
   - 逻辑层变更后，通过事件通知 UI 刷新，是稳定结构。

4. **配置数据与逻辑分离**
   - 关卡参数进 JSON，玩法逻辑在 TS，后续调难度无需改代码。

5. **提供兜底数据，保证永不白屏/卡初始态**
   - 即使资源加载失败，也能进入内置关卡，便于定位问题。

## 4.2 坏问题（踩过的坑）与规避方式

1. **坑：`assets` 脚本依赖仓库外 `src/*` 路径**
   - 现象：Cocos 报 module not found / extension 错误。
   - 规避：运行所需脚本必须全在 `assets/scripts` 可解析范围内。

2. **坑：只有静态 UI，没有动态渲染链路**
   - 现象：黑底 + 按钮，但游戏不动。
   - 规避：必须保证 `GameController -> Event -> UI refresh` 整链路可达。

3. **坑：组件赋值时机与订阅时机错位**
   - 现象：状态停留在“初始化中”。
   - 规避：UI 脚本支持 `setup()` 迟绑定，启动器在关键阶段显式调用。

4. **坑：resources 加载失败导致无法开局**
   - 现象：卡在初始化文案、无牌面。
   - 规避：加 fallback level，先开局再尝试加载 60 关。

5. **坑：编辑器缓存导致“改了代码像没生效”**
   - 现象：行为与源码不一致。
   - 规避：删除 Cocos 项目 `temp/` 和 `library/` 后重开。

## 5. 标准排障流程（遇到“静态页面”时）

按顺序执行，不要跳步：

1. 看 `Canvas` 是否挂了 `QuickStartLauncher`。
2. 看 `assets/resources/config/levels.json` 是否存在且为 JsonAsset。
3. 运行后看状态文案：
   - 若显示“使用内置关卡”：说明 resources 加载失败，但逻辑已可玩，优先查资源导入路径。
4. 清理缓存：删除 `temp/`、`library/`，重开工程。
5. 检查脚本是否有外部路径引用（例如 `../../../src/*`）。

## 6. 代码与配置变更 SOP（团队协作版）

### 6.1 修改玩法逻辑（TS）
1. 在 `assets/scripts/core` 修改。
2. 保持 `GameController` 对 UI 的事件通知不破坏。
3. 运行冒烟测试（能开局、能点牌、能重开）。

### 6.2 修改关卡难度
1. 修改 `config/levels.json`（或用脚本重生）。
2. 同步到 `assets/resources/config/levels.json`。
3. 运行验证前 3 关 + 中段 1 关 + 末段 1 关。

### 6.3 修改启动流程
1. 优先在 `QuickStartLauncher` 做增量。
2. 保留 fallback 逻辑，不要删除。
3. 保留最简挂载：`Canvas` 挂 1 个脚本即可运行。

## 9. 一句话原则

> **先保证“任何情况下都能开局可玩”，再追求完整体验。**

这条原则能显著降低联调成本，避免你再次遇到“界面能打开但游戏不动”的问题。

## 10. 项目全过程复盘（里程碑时间线）

1. **阶段 0：脚手架搭建**
   - 完成基础目录、核心脚本、关卡 JSON、文档模板。
   - 教训：只“有代码文件”不等于“Cocos 可运行”，必须验证编辑器实际挂载链路。

2. **阶段 1：自包含改造**
   - 去除 `assets/scripts` 对仓库外 `src/*` 的硬依赖，保证仅复制 `assets/` 也可运行。
   - 教训：Cocos 项目实际导入范围决定模块解析边界，跨目录引用容易在用户工程中失效。

3. **阶段 2：一键启动（QuickStart）**
   - 引入 `QuickStartLauncher`，把“手动拖节点、绑组件、配按钮”改为运行时自动装配。
   - 教训：低手工是稳定交付前提，尤其面向非程序同学时。

4. **阶段 3：静态页/初始化卡死排障**
   - 通过 `setup()` 迟绑定 + fallback level + 明确状态文案，修复“状态：初始化中”不动问题。
   - 教训：组件赋值时机和事件订阅时机是 Cocos UI 常见坑，必须设计幂等绑定。

5. **阶段 4：关卡推进与交互回归修复**
   - 修复通关后无法进下一关、点击被拦截、按钮可见性不稳定等回归。
   - 教训：输入链路（Button/Touch）改动风险高，新增交互要先做回归清单。

6. **阶段 5：资源接入（图片/槽位/失败弹层）**
   - 补齐宠物图加载路径兼容、槽位图标显示、失败弹层可读性与重开可用性。
   - 教训：资源导入参数和运行时路径必须双向对齐（Importer + `resources.load` 路径）。

7. **阶段 6：移动端适配增强**
   - 提升竖屏利用率：HUD 紧凑化、牌盘放大、列数与尺寸调整（当前固定 7 列）。
   - 教训：PC 看起来正常 ≠ 手机可读，必须按真实竖屏目标做布局预算。

8. **阶段 7：体验补齐（BGM）**
   - 增加 `assets/resources/audio/bgm/bgm.mp3` 自动循环播放。
   - 教训：音频也需要“可缺省兜底”，加载失败不能影响主流程。

## 11. 可复用的“最优提示词”与执行过程（下个同类项目直接套用）

### 11.1 最优提示词（可直接复制）

```text
你是资深 Cocos Creator 3.8.2 工程师。请按“最少手工接入”目标交付一个可运行的三消+槽位原型：

硬性要求：
1) 只需在 Cocos 主场景给 Canvas 挂 1 个脚本即可运行；
2) assets/scripts 必须自包含，禁止依赖仓库外路径；
3) resources 加载失败时必须自动 fallback 到内置关卡，不能卡在初始化页；
4) 提供完整闭环：点牌→进槽→三消→胜利/失败→重开/下一关；
5) 手机竖屏优先，最大化可用空间并保证卡牌可读；
6) 图片资源与音频资源给出明确目录规范与运行时加载路径；
7) 输出：代码变更 + 操作步骤 + 排障清单 + 回归测试命令。

验收标准：
- 首次接入 5 分钟内可跑；
- 无资源时仍可玩；
- 有资源时自动显示图片与播放 BGM；
- 第 1 关与第 60 关均可验证；
- 失败弹层按钮可点击，交互无阻断。
```

### 11.2 最优开发过程（8 步）

1. **先定义“单脚本启动”边界**：先写 `QuickStartLauncher` 骨架，确保不依赖编辑器手工节点。
2. **先做 fallback 可开局**：先内置 1 关，确保任何情况下能点牌可玩。
3. **再接 60 关资源加载**：`resources.load` 成功后热切换真实数据。
4. **再接 UI 订阅链路**：统一 `setup()/bindIfPossible()`，避免时序问题。
5. **再接图片/音频资源**：路径、命名、Importer 参数一起定义并文档化。
6. **再做移动端排版**：以手机竖屏为主，固定关键约束（如列数、最小尺寸）。
7. **最后补排障文档**：把“症状→原因→修复步骤”写成 checklist。
8. **每次改交互都跑回归**：至少覆盖“可点击、可重开、可进下一关、可失败恢复”。

### 11.3 长期记忆（以后必须遵守）

- 规则 1：**先保证可玩，再保证好看**。
- 规则 2：**任何资源加载都要有兜底**。
- 规则 3：**新增交互必须做回归，防止点击链路被误伤**。
- 规则 4：**移动端布局独立预算，不拿 PC 效果当标准**。
- 规则 5：**文档必须包含“最小接入步骤 + 排障清单 + 验收命令”**。
